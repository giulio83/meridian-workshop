Technical Approach

How we'll do it

1

Codebase audit first

Before writing a line, we'll map what the previous vendor built against what the handoff says. We expect gaps — we'll find them before they become surprises mid-engagement.

2

Test infrastructure as foundation

R3 ships in Phase 1, not Phase 3. Playwright end-to-end tests go in before we touch the Reports bugs — so every fix is verified and IT can approve changes from day one.

3

Fix, then extend

R1 (Reports) before R2 (Restocking). A clean, well-tested foundation makes the new feature faster to build and safer to ship. No new features on broken ground.

4

Desired features scoped in, not promised

D1–D3 are included in the timeline with explicit decision points. We commit to the required work; the desired work gets scheduled based on real progress, not optimistic estimates.
